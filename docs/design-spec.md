# Cocopua HP 設計書

最終更新: 2026-04-22

## 1. 目的

- アイサロンの魅力を伝え、予約行動に迷いなく到達できるサイトを構築する。
- モバイルユーザー中心で、短時間で必要情報を得られる情報設計を行う。
- 高コストな独自開発を避け、運用しやすいMVPを先行リリースする。

## 2. スコープ

### 2.1 対象

- 公開サイト（サロン紹介）
- お知らせ機能（一覧・詳細）
- 管理向けお知らせ投稿画面（投稿・編集・削除・公開管理）

### 2.2 対象外

- 独自予約システムの開発
- 会員登録・決済・ポイントなどEC機能
- 高度な分析ダッシュボード

## 3. 技術構成

- フロントエンド: React + TypeScript + Vite
- UI: Tailwind CSS + shadcn/ui
- ルーティング: React Router（導入予定）

## 4. 画面設計

## 4.1 公開画面

- `/` Top
- `/menu` Menu
- `/gallery` Gallery
- `/staff` Staff
- `/access` Access
- `/faq` FAQ
- `/news` News一覧
- `/news/:id` News詳細

### 画面共通要件

- ヘッダーにロゴと主要導線を配置する。
- フッターに店舗情報（住所・営業時間・最寄り駅）を表示する。

## 4.2 管理画面

- `/admin/news` お知らせ一覧（編集・削除・公開状態）
- `/admin/news/new` 新規作成
- `/admin/news/:id/edit` 編集

### 管理画面要件

- 一覧で公開/非公開が判別できる。
- 編集画面で公開状態を切り替え可能。
- 保存後は一覧へ戻り、状態が即時反映される。

## 5. データ設計（お知らせ）

## 5.1 News 型

```ts
export type NewsItem = {
  id: string
  title: string
  body: string
  published: boolean
  publishedAt: string | null
  updatedAt: string
}
```

## 5.2 表示ルール

- 公開画面: `published === true` のみ表示する。
- 管理画面: すべて表示し、状態でフィルタ可能にする。
- 一覧順: `publishedAt`（公開済み）または `updatedAt` の降順。

## 6. コンポーネント方針

- 共通UIは `src/components` 配下へ配置する。
- 機能固有は `src/features/news` に閉じ込める。
- 画面は `src/pages/public` と `src/pages/admin` で分離する。

## 7. 非機能要件

## 7.1 アクセシビリティ

- 画像に代替テキストを設定する。
- 見出し階層を論理的に保つ。
- フォームラベルと入力要素を関連付ける。
- 主要操作はキーボードで実行可能にする。

## 7.2 パフォーマンス

- ロゴ・画像は適正サイズで配信する。
- 不要ライブラリを増やさない。
- 主要ページのLCP悪化を防ぐ。

## 7.3 SEO

- 各ページに固有title/descriptionを設定する。
- 店舗情報を明記してローカル検索流入を意識する。

## 8. 受け入れ基準

- お知らせ管理画面でCRUD+公開管理が可能。
- 公開/非公開の切替が公開画面に反映される。
- スマートフォン幅（360px-430px）でUI崩れがない。

## 9. 実装フェーズ

1. ルーティングと公開側ページ骨組み実装
2. 共通ヘッダー/フッター/予約CTA実装
3. お知らせ機能（公開側）実装
4. お知らせ投稿画面（管理側）実装
5. コンテンツ反映と最終調整
