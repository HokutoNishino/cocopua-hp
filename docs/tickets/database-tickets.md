# データベース実装チケット（MySQL 8.4 LTS）

最終更新: 2026-04-29

## 運用ルール

- 優先度: P0 > P1 > P2
- 見積: S（0.5日）, M（1日）, L（2-3日）
- MVPではWordPress標準テーブル運用を優先し、独自テーブル追加は原則しない

## P0（最優先）

### DB-001 MySQL 8.4 LTS 環境確立
- 目的: 安定版DBでの運用基盤を作る
- 作業:
  - MySQL 8.4 LTSを構築
  - WordPress接続確認
- 完了条件:
  - WordPressがDB接続して正常起動
- 見積: S
- 依存: なし

### DB-002 文字コード・照合順序統一
- 目的: 日本語データの文字化けを防ぐ
- 作業:
  - utf8mb4 / utf8mb4_unicode_ci を適用
- 完了条件:
  - 主要テーブルが統一設定
- 見積: S
- 依存: DB-001

### DB-003 最小権限ユーザー作成
- 目的: セキュリティリスクを下げる
- 作業:
  - アプリ接続ユーザーを最小権限で作成
- 完了条件:
  - 管理ユーザーと分離されている
- 見積: S
- 依存: DB-001

## P1（高）

### DB-004 バックアップ自動化
- 目的: 障害時の復旧性を担保する
- 作業:
  - 日次フルバックアップ
  - 保持14日以上
- 完了条件:
  - 定期バックアップが自動実行される
- 見積: M
- 依存: DB-001

### DB-005 リストア手順検証
- 目的: RTO/RPOを実運用で満たせるか確認する
- 作業:
  - リストア演習を実施
  - 手順書を更新
- 完了条件:
  - 復旧時間4時間以内を確認
- 見積: M
- 依存: DB-004

### DB-006 スロークエリ監視
- 目的: 性能劣化の早期検知
- 作業:
  - スロークエリログ有効化
  - 通知条件を設定
- 完了条件:
  - 閾値超過時に通知される
- 見積: S
- 依存: DB-001

## P2（中）

### DB-007 インデックス見直し基準策定
- 目的: 過剰チューニングを避けつつ性能を維持する
- 作業:
  - API遅延時の調査手順を定義
  - 追加インデックス判断基準を文書化
- 完了条件:
  - 調査から対応までの判断フローがある
- 見積: S
- 依存: DB-006

### DB-008 容量監視とアラート
- 目的: ディスク逼迫の予防
- 作業:
  - 使用率の監視閾値設定
  - アラート通知連携
- 完了条件:
  - 容量逼迫前に通知される
- 見積: S
- 依存: DB-001
