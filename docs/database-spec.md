# Cocopua データベース設計書

最終更新: 2026-04-29

## 1. 採用方針

- DBMSは **MySQL 8.4 LTS（安定版）** を採用する。
- 文字コードは `utf8mb4`、照合順序は `utf8mb4_unicode_ci` を基本とする。
- WordPress標準テーブルを主軸に運用し、初期MVPでは独自テーブルを増やさない。

## 2. バージョン・互換性

- MySQL: 8.4 LTS
- WordPress: 6.0+
- PHP: 8.1+

## 3. 論理データ設計

### 3.1 コンテンツ

- News: `wp_posts`（post_type = news）
- 固定ページ（Top）: `wp_posts`（post_type = page）
- カスタムフィールド: `wp_postmeta`

### 3.2 メディア

- 画像: `wp_posts`（post_type = attachment）
- 付随情報: `wp_postmeta`

## 4. 物理設計

### 4.1 テーブル方針

- 使用テーブル（主要）:
  - `wp_posts`
  - `wp_postmeta`
  - `wp_terms`
  - `wp_term_taxonomy`
  - `wp_term_relationships`
  - `wp_users`
  - `wp_usermeta`

### 4.2 インデックス方針

- WordPress標準インデックスを基本利用
- API遅延が目標を超える場合のみ、クエリ分析後に追加検討

## 5. 性能要件

- News一覧APIのレスポンス: 1秒以内（通常時）
- 同時アクセス時もタイムアウト率を低水準維持

## 6. バックアップ/リカバリ

### 6.1 バックアップ

- DB: 日次フルバックアップ
- 保持期間: 14日以上
- メディア: 週次バックアップ

### 6.2 リカバリ

- 目標復旧時間（RTO）: 4時間以内
- 目標復旧時点（RPO）: 24時間以内

## 7. セキュリティ

- DB接続は最小権限ユーザーを利用
- 本番DBは外部公開しない
- 管理接続元IPを制限

## 8. 監視

- CPU/メモリ/接続数を監視
- スロークエリログを有効化
- しきい値超過時に通知

## 9. 将来拡張

- 予約データを扱う場合は別テーブルを分離
- 分析用途はレプリカまたはDWHへ分離

## 10. 受け入れ基準

- MySQL 8.4 LTSでWordPressが安定稼働する
- News/Topの読み取り性能要件を満たす
- バックアップとリストア手順が実行可能
